# Local Naïve-100 Side-by-Side Comparison

We are currently running a background script to execute the `naive-100` order and the optimized order (e.g. `pkg-rt-front`, `alloc-sort`) side-by-side on our local machine for all measured projects. This is necessary to compute a scientifically rigorous "% Improvement vs Naïve-100" metric.

Our initial approach was to normalize across the two different hardware platforms by comparing the remote `naive-100` percentage speedup (vs remote initial) against the local optimized percentage speedup (vs local initial). However, a quick local verification on `javaparser-core-testing` revealed this cross-machine normalization is highly misleading. While `naive-100` achieved a 17.1% speedup on the remote server, it scaled up massively to a 26.8% speedup when run locally on the Mac. By improperly subtracting the remote percentage from the local optimized percentage (23.3%), we incorrectly concluded that the optimized strategy was 6.2 percentage points *faster* than `naive-100`, when in reality it was 8.5 percentage points *slower* when both were run locally. This hardware scaling discrepancy proves that we must run both orders on the exact same hardware to draw valid conclusions, which is exactly what our active background script is doing.
