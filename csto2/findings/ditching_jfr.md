# Ditching JFR: Rationale and Findings

We have completely removed Java Flight Recorder (JFR) tracing and JFR-based candidate strategies (`jfr-gc-front`, `jfr-warmup-front`, `pairwise-warm` causal confirmation) from the CSTO2 optimization portfolio. Dynamic test profiling is now powered exclusively by standard JVM Management MXBeans (`CompilationMXBean` and `ThreadMXBean`), which capture JIT compilation time and allocated memory bytes at test-class boundaries.

This decision was driven by empirical results showing that JFR-based strategies provided zero practical optimization. In all target modules benchmarked, MXBean-driven strategies (such as `alloc-front`, `jit-sort`, and package-level runtime/allocation sorting) consistently matched or outperformed JFR strategies. Removing JFR eliminates the continuous 2% to 15% execution profiling overhead during trace runs, avoids large binary file writes and out-of-process JSONL parsing, and ensures 100% JVM compatibility and platform reliability across different JDK distributions without relying on HotSpot-specific native libraries.
