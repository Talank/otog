# CSTO v2 (`csto2`) Log & Artifact Guide

This guide describes the various logs and data files generated by the CSTO v2 pipeline, explaining where they are located, what they contain, and how to use them for debugging.

---

## 1. Directory Layout (Host vs. Container)

When running the pipeline via Docker, your host directory `/home/gcohen4/otog/results` is mapped directly to `/workspace/.csto2` inside the container. 

```text
/home/gcohen4/otog/results/              <-- Host path (corresponds to /workspace/.csto2)
├── config.properties                   # Persisted run configuration (Git URL, commits, JVM paths)
├── deps.classpath                      # Generated target classpath entries
├── tests.all                           # List of all classes discovered in the module
├── tests.runnable                      # Filtered list of runnable test classes (concrete, non-abstract)
├── trace/                              # Directory for the tracing/calibration phase
│   ├── trace.jsonl                     # Compiled master trace file (Surefire + Agent facts)
│   ├── orders/                         # Test orders sent to Surefire
│   │   ├── initial.order
│   │   └── shuffle-1.order (up to N)
│   ├── agent/                          # Raw JVM metrics from the Java Agent
│   │   ├── initial.facts.jsonl
│   │   └── shuffle-1.facts.jsonl (up to N)
│   └── logs/                           # Raw stdout/stderr console outputs from Maven Surefire
│       ├── initial.log
│       └── shuffle-1.log (up to N)
└── select/                             # Directory for the selection phase (once selection begins)
    ├── candidates/                     # Test orders for candidate strategies
    └── logs/                           # Timing run logs for A/B validation
```

## 2. Log Types & Detailed Breakdown

### Type A: Raw Maven Surefire Console Logs
* **Location (Host)**: `results/trace/logs/*.log` (e.g., [initial.log](file:///home/gcohen4/otog/results/trace/logs/initial.log), [shuffle-1.log](file:///home/gcohen4/otog/results/trace/logs/shuffle-1.log))
* **Format**: Plain Text
* **What it contains**: 
  The raw console output of the Maven command run for that order:
  ```bash
  mvn initialize surefire:test -Dmaven.ext.class.path=... -Dsurefire.runOrder=testorder -Dtest=...
  ```
  It captures Maven build steps, compiler warnings, Java process prints (`System.out`), and standard Surefire execution logs showing the start and result of each test class.
* **When to read it**:
  * If a run exits with a failure (`exit=1`), open these logs immediately.
  * Search for `BUILD FAILURE`, `PluginResolutionException`, `UnsupportedClassVersionError`, or stack traces from crashing tests.

---

### Type B: Master Trace Compilation File
* **Location (Host)**: `results/trace/trace.jsonl` (e.g., [trace.jsonl](file:///home/gcohen4/otog/results/trace/trace.jsonl))
* **Format**: JSON Lines (Flat JSON objects per line)
* **What it contains**:
  A merged record of Surefire's XML report metrics combined with the JVM agent's metrics. Each row represents a test class execution:
  ```json
  {"orderId":"initial","position":0,"test":"io.netty.bootstrap.BootstrapTest","runtimeMs":6432.0,"testsFound":16,"failures":0,"status":"PASS","classesLoaded":1042,"jitMs":1402,"gcCount":2,"gcMs":82,"allocBytes":248901004,"threadDelta":0}
  ```
* **Key Fields**:
  * `runtimeMs`: Wall-clock execution time for this test class.
  * `status`: `PASS` or `FAIL`. (All must be `PASS` across all orders for the trace to be valid).
  * `classesLoaded`/`jitMs`/`allocBytes`: Durations/counts compiled by the `-javaagent`.
* **When to read it**:
  * To inspect the parsed timings and JVM overhead metrics for your classes.
  * To verify that all runs successfully generated metadata rows.

---

### Type C: Raw Agent Instrumentation Facts
* **Location (Host)**: `results/trace/agent/*.facts.jsonl`
* **Format**: JSON Lines
* **What it contains**:
  The raw per-class JVM measurements recorded by the JFR/MXBean instrumentation agent (`csto2-agent.jar`) before they are merged into the master trace. These are written directly inside the forked JVM.

---

### Type D: Active Workspace Configuration
* **Location (Host)**: `results/config.properties` (e.g., [config.properties](file:///home/gcohen4/otog/results/config.properties))
* **Format**: Properties file (`key = value`)
* **What it contains**:
  The active run settings used to launch the Docker compiler and JVM runner:
  ```properties
  repo_url = https://github.com/netty/netty.git
  commit_sha = 57bea3bea22717639ba200432c81b23154e4bbd9
  module_dir = transport
  java = /usr/lib/jvm/java-11-openjdk-amd64
  mvnopts = -Djacoco.skip=true ...
  ```
* **When to read/edit it**:
  * To inspect what compile arguments (`mvnopts`) are currently active.
  * To manually override options (e.g., pointing to custom Java paths or adding plugin exclusions).

## 3. How to Debug Common Failures

### 1. `trace produced no rows — every order's Surefire run failed (no reports)`
* **The Symptom**: The dynamic trace crashes immediately with `java.lang.IllegalStateException: trace produced no rows`.
* **The Cause**: The individual Maven executions crashed before running any tests.
* **The Fix**: Open `results/trace/logs/initial.log`. It is usually a Maven configuration problem, compilation error, or missing dependency (e.g., the `3.0.0-M8-SNAPSHOT` Surefire plugin error).

### 2. `Corrupted channel by directly writing to native stream`
* **The Symptom**: A warning in the raw log saying: `Corrupted channel by directly writing to native stream in forked JVM`.
* **The Cause**: A test class or library is printing binary/non-UTF8 characters directly to `System.out` or `System.err`, which corrupts Surefire's fork communication channel.
* **The Fix**: Usually benign if the build finishes with `BUILD SUCCESS`, but if it causes a crash, check which test was running immediately prior to the warning and check its output.

### 3. Test Flakiness (Red Tests)
* **The Symptom**: A shuffle run fails with `exit=1` due to test failures, and the trace fails the "green gate."
* **The Cause**: Some tests are order-dependent (they pollute static state that subsequent tests read).
* **The Fix**: Search for `[INFO] Running ...` followed by `Failures: X` in the failing `shuffle-N.log` to identify the failing test and trace its class dependency order in the corresponding `results/trace/orders/shuffle-N.order` file.
